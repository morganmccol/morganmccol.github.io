Automating Releases with GitHub

Posted 1. February 2020. 2 min read.

Step-by-Step

Step 1: Release CII

Release refers to freezing code using Git tags, setting up the staging database, and creating a setup executable.

  1. Run the release script for v2.6.0 on the following repositories:

    • docker
    • genetesis-api
    • cardioflux-control-service
    • process-pipeline-service
    • scan-data-service
    • pdf-service
    • shared-ui
    • site-reader-portal
    • operator-portal
  2. Use generated changelogs to publish release notes
  3. Backup production and staging databases
  4. Overwrite staging database with production database
  5. Run migrations on staging database

Step 2: Deployment Clinical Sites

Deployment refers to installing released software on a CardioFlux device at a clinical site.

2.1: Staging

  1. Install v2.6.0-staging software.
  2. Test using staging database.
  3. Once all sites are confirmed to work with staging, continue to perform production deployment.

2.2: Production

  1. Run migrations on production database
  2. Install v2.6.0-production software
  3. Test using production database
  4. Remove v2.6.0-staging software
  5. Remove v2.5.0-production software

Installation Manual

Running Automatic

Deployment

Release Rollout

Staging

  1. Backup prod and staging database
  2. Copy prod database to staging database, overwriting it
  3. Run migrations on staging database
  4. Visit each site, pull new code, and test using staging database. Keep old production code working if possible.
  5. If successful, go to the next site and repeat step 4

Production

  1. Once all sites work with staging, prepare the sites for some downtime while we update everyone
  2. Run migrations on prod database
  3. Remote into each site and pull new code, pointing to prod database
  4. Test code to ensure everything works
  5. If successful, perform cleanup remove old code, test new code and go to the next site and repeat steps 3-5.
